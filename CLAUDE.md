# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

**Dominion** is a live territory-control simulation: four powers compete for land each "cycle"; the user can pause/step and adjust values to intervene while it runs.

- **Two editions share one simulation.** `dominion.py` holds **the simulation** (the `cycle()` pipeline, the lord, races, the stat/journey system) *and* a stdlib **curses** front-end. `dominion_app.py` is the **pygame** front-end and is **the runtime the player uses** (`python3 dominion_app.py`); it does `import dominion as d` and reuses the sim, but has its **own** rendering, input loop, music, boss-fight, and UI (character window, recruit modal, charged bolt, etc.). **So: simulation/game-logic changes go in `dominion.py` (both editions get them); player-facing UI/input/render changes go in `dominion_app.py`.** The curses UI in `dominion.py` is the legacy edition — keep it runnable, but it is *not* what the player runs, so it can lag behind on UI features. ⚠️ Editing the shared sim can break the pygame app if it constructs sim inputs in an old shape (e.g. the seeker dict) — update both. (There was also once a browser front-end `index.html`, scrapped.)
- `chiptune.py` — a small, optional, pure-stdlib helper (not a second edition of the sim): a MIDI parser + square-wave synth that bakes a `.mid` into a `.wav` so background music can play (this box has no MIDI synth/soundfont). `dominion.py` imports it lazily for the music feature; the sim runs fine without it.
- `music/` — drop a `.mid`/`.midi` (baked to chiptune on first run, cached as a sibling `.wav`) or a ready `.wav` here; the game loops the first track it finds. Playback shells out to `paplay`/`aplay` (the only audio dependency, used only if present); **muted by default — toggle (mute/unmute) with `p`**. No track / no player → silent, no error.

`DESIGN.md` is a forward-looking design sketch ("Living World" / wizard-inside-the-world LOD concept). **Nothing in it is implemented** — the current code is only the macro slider sim. Treat it as future direction, not a description of the codebase; don't infer that any of its mechanisms exist.

### Running

- **Pygame edition (what the player runs): `python3 dominion_app.py`** (needs `pygame`; `pip install -r requirements.txt` covers the one dependency).
- Legacy terminal edition under a memory/CPU cap: `./run.sh` (wraps `dominion.py` in a `systemd-run --user --scope` cgroup with `MemoryMax`/`CPUQuota`/`MemorySwapMax=0`, so a runaway is OOM-killed instead of taking down the machine). Override caps via env: `MEM_MAX=512M CPU_QUOTA=200% ./run.sh`.
- Legacy terminal edition directly (no cap): `python3 dominion.py`.
- **Smoke-testing the pygame app headlessly:** `SDL_VIDEODRIVER=dummy SDL_AUDIODRIVER=dummy python3 -c "import dominion_app as A, pygame; g=A.Game(); g.update(0.016); g.render()"`, and drive input by constructing fake events (`type=pygame.KEYDOWN/KEYUP`, `.key=...`) into `g.on_key`/`g.on_keyup`. The sim itself is still exercised headlessly via the `dominion.py` curses-stub recipe below.

There is nothing to build or lint. There is no test suite, but the simulation logic in `dominion.py` is pure and can be exercised headlessly: stub `curses` (it's only imported for color constants and the TUI), then drive `fresh_state()` / `cycle()` and assert the invariants below. For example:

```sh
python3 -c "
import sys, types
cur = types.ModuleType('curses'); cur.__getattr__ = lambda name: 0; sys.modules['curses'] = cur
import dominion as d
s = d.fresh_state()
for _ in range(2000):
    d.cycle(s)
    assert abs(sum(s['territory']) - 100) < 1e-6                  # territory sums to 100
    assert all(t >= d.MIN_TERR * 0.95 for t in s['territory'])    # soft floor (see note)
    assert all(0 <= a <= 100 for a in s['army']) and len(s['items']) <= d.MAX_ITEMS
    assert all(s['hearts'][i][j] == s['hearts'][j][i] for i in range(d.N) for j in range(d.N))
    c = d._counts(s['owner'])
    assert sum(c) == d.TILES and all(x >= 1 for x in c)             # map tiles conserved, no power wiped off it
    assert c == d.tile_targets(s)                                   # map matches territory exactly (0 drift)
print('ok')
"
```

The `curses` stub must answer to any attribute (`dominion.py` reads `curses.COLOR_*` at import time), hence the `__getattr__` module trick above.

## Design philosophy

**Infer, don't add.** Prefer deriving new behavior from the variables that already exist (`territory`/the tile map, `army`, `hearts`, `items`, `battles`) over introducing a new faction variable. Before adding state (wealth, population, …), apply the **orthogonality test**: does it let power *diverge* from what the current variables already determine — a small-but-strong or large-but-hollow power? If yes, it earns its place. If it would move in lockstep with something we already track (e.g. wealth ∝ army ∝ land), it's redundant — infer it on demand instead. A new variable must pay for itself in dynamics that can't otherwise occur.

**The macro % is the source of truth; the wizard view folds out of it.** The abstract sim figures — territory share, corruption %, a quest's find-odds / difficulty, favor, army, etc. — are the **first port of call** when deciding what the wizard-level (micro / realtime) view should show or do. Wherever possible the lived view should be a **derivation** of those macro numbers, not its own independent truth that floats free of them: a quest's difficulty % *becomes* the monster/miniboss density you fight when escorting its seeker; a tile's corruption % *becomes* its menace (fallen → infested); `territory[]` *becomes* the world map; a battle's % progress *becomes* the besiegers drawn on the tile. So when adding a micro/visual mechanic, first ask "which macro number is this expressing?" and unfold it from there — don't invent parallel micro-state for something a macro figure already determines. ("Procedurally generate" isn't quite the term — the point is the macro value is upstream and the detailed experience unfolds downstream from it.) Break this only when no macro derivation is genuinely possible.

## Architecture

`dominion.py` is organized in labeled sections that form a clear data → simulation → render pipeline:

- **Configuration** — `POWERS` (the four factions), `ITEM_TYPES` (rarity-weighted magical relics), and `status_of(hearts)` which maps a 0–10 relationship meter to ALLIED/PEACE/ENEMIES/AT WAR. Each faction is a fixed coloured **realm** (`REALMS` — colour + biome, stable per index) peopled by a **race** (`RACES` = Elves/Men/Dwarves/Cyclopes). The race→realm pairing is rolled fresh each run by `assign_races()` (called from `fresh_state`), which stores the order in `state["race"]` and rebuilds `POWERS`' display names (`"<Realm> <Race>"`) and `FACTION_EMOJI` — so colour/biome identity is stable while who-is-what is random. Race's only effect so far is a **bias on stat rolls** (see Stats); each will grow its own state later.
- **Stats** — one shared schema for every actor (wizard, lord, recruits, the anonymous crowd): four 1–10 aptitudes `STATS` = **Might / Endurance / Wits / Swiftness**. They drive *tasks* (a relic journey weights them — `relic_demand`/`journey_fit`/`seek_rate`, see Seekers) and *arena combat* as **threshold unlocks** (`WIZ_ABILITIES`): a stat past its gate grants a new verb rather than a bigger number — Might→a **charged heavy bolt** (`WIZ_MIGHT_CHARGE_AT`, wired: **hold space** to wind it up, release to loose a piercing bolt that kills every minion in its path; a tap is still a basic bolt; moving cancels. "Hold" is inferred from key auto-repeat since curses has no key-release, so `SPACE_HOLD_GRACE` rides out a slow repeat delay — a caveat being that mashing space at high Might also charges), Wits has **two tiers** — `≥ WIZ_WITS_SEEK_AT` (4) to *track your own seekers' progress/ETA* in the HUD, `≥ WIZ_WITS_SENSE_AT` (7) to *sense the shadow* (the lord shows on the world map); both wired (pygame). Swiftness→swifter movement (`WIZ_SWIFT_FAST_AT`, pending); Endurance is the exception, scaling **max hearts** continuously (`wiz_max_hearts()` = the Endurance stat, wired — the lord/boss will get the same unlocks from its own stats, minus the map sense). `roll_stats(race)` rolls a block biased by the race's `RACE_APTITUDE` (a lean, not a lock — ranges overlap). This layer is **wizard-level only — it never touches the macro sim** (territory/army/hearts). Actor blocks: `state["wiz_stats"]`, `lord["stats"]` (rolled + boosted), and transient crowd blocks from `roll_townsfolk`. The **character window** (`c` in the tile view, rendered in `draw_tile`, keys handled in `main`) sets the wizard's stats by hand with **no limits** (a testing bench) and lists each stat's unlock + locked/UNLOCKED status.
- **Knowledge of the world** (the wizard starts **blind** — `state["map_snapshot"]`/`map_live`, pygame `render_map`): there is no world map until he **charts one in a town** (`chart_world` on entering a peaceful town snapshots borders/corruption/territory/army/shadow + the day). The map view then renders that **frozen snapshot** with a staleness readout ("charted day X (Yd stale)"); it only refreshes by returning to a town. His own position is always live; the lord still shows via the Wits sense. **The POWERS map (faction territory/army/corruption) is always fully updated by any chart** — but the **GEOGRAPHY map is pieced together gradually**: each chart only uncovers a circle of terrain around the town it was bought in (`d.MAP_REVEAL_RADIUS` tiles; a capital's chart reveals twice that radius, `MAP_REVEAL_RADIUS_CAPITAL`), accumulated across visits in `state["map_revealed"]` (a per-tile revealed bitmask, unioned with each new chart, never cleared) — so the geography view fogs out anything never charted near, and filling it in means visiting many different towns. `render_world_geo` (pygame) draws this as a cached fog overlay (`bake_fog_surf`) over the always-fully-baked terrain Surface. **Recruits show on the map only while "on their path"** (`q["on_path"]`, re-rolled each cycle = `random() < fit` for seekers, always true for marchers/fighters heading to a known front): a well-suited (high-fit) agent keeps its route and is findable on the map; a poor fit **strays** (pursued/lost/attacked) and vanishes — then you must **ask around** for a bearing (below). So suitability-to-task governs both speed *and* how reliably you can locate them. **`l`** toggles a live (always-current) map for testing. **Relics are also unknown until their legend is heard:** `state["relics_known"]` starts empty, and the wizard can only send seekers after relics he's learned of. The recruit modal offers only known relics. (The lord is unaffected — he knows the world and seeks any relic.)
- **Talking — the chat window** (`e` near anyone; `self.chat` + `handle_chat`/`render_chat`, pygame): walk up to a **townsperson or a wandering/fishing folk out in the country** and ask — `[1]` **old legends** (only a town **elder** with an untold tale knows one — `town_elders()`, deterministic per town, gold-dot-marked; an elder's single tale is drawn from a combined pool of **faction relics and artifact-quest creatures** — `("relic", type)` learns a relic exists, adding it to `relics_known`; `("artifact", index)` instead sets that artifact's `"known"` flag, a rumor of the creature's existence distinct from `"told"` — see Artifact quests below), `[2]` **your agents** (each recruit's bearing — cardinal N/S/E/W from you + range close/"a way off"/far), `[3]` **news of the world** (where the shadow's hand lies heaviest; any war), `[4]` **way to the nearest town** (`nearest_town_dir()` — the nearest village/city's name + a cardinal bearing incl. diagonals + range band; recognises when you're already in one). Answers fill in below; any other key leaves. (There is **no "say hello for favor"** anymore — chatting is for information; favor moves only by deeds.)
- **Equipment (wizard gear)** — first slice: an **equipped staff** (`state["wiz_staff"]` = index into pygame `STAVES`), the wizard's *attack kit*, distinct from faction relics. A staff defines its shot pattern (`shots` = angle offsets fired per attack) and `power_mult`, and gates equip on Might (`equip_might` — "magic" is folded into **Might for now**, to fork into its own stat later). `Game.fire_bolt` fires the equipped staff's pattern for both basic and charged shots. **`v`** cycles to the next staff the wizard has **found** *and* his Might can wield; the character window shows the equipped staff; dropping Might below the requirement auto-reverts to the Plain Staff. Staves beyond the Plain Staff (idx 0, always owned) are **won from artifact quests** — `state["staves_found"]` (a set of STAVES indices) gates `cycle_staff`. (Planned: clothes/trinket slots giving buffs; staves with spell trade-offs.)
- **Artifact quests (wizard gear sought in person)** — some gear isn't bought or handed out but **adventured for** by the wizard himself (his own quest, distinct from the faction relics recruits chase). `place_artifacts(state)` (dominion.py, from `fresh_state`) hides each in `state["artifacts"]` — one entry per `ARTIFACT_SPECS` item `{grants, staff, name, creature, creature_name, hint, loc, known, told, found}`: a **mythical creature** dwells at a wild `creature` tile and the prize rests at a wild `loc` tile (all creatures/locs walkable, clear of towns, spread apart; deterministic per `world_seed`). Each artifact `grants` a **staff** (`grants:"staff"` + `staff` index), an **ability** (`grants:"charge"`), or **worn gear** (`grants:"cloak"`). Three ship today, each told by its **own** creature: the **Fanstaff** (3-way spread staff, Might 6 — *the Sphinx of the Wastes*), **the Charged Bolt** (the wound-up heavy/piercing bolt — *the Drake of the Embers*), and **the Veilcloak** (halves harm taken — *the Weaver of the Hollow*). So the charged bolt is **no longer a pure Might unlock**: `state["can_charge"]` starts False and the space-charge handler requires it (plus Might ≥ `WIZ_MIGHT_CHARGE_AT`). The cloak (`state["has_cloak"]`) routes all wizard damage through `Game.hurt()`, which **turns aside every other blow** (a parity counter, `self.dmg_parity`) — exactly half harm — with a "cloak turns it!" toast; all three damage sites (monster touch, monster shot, boss/guardian touch) call `hurt()`. The character window shows the equipped staff + cloak status. **`"known"` vs `"told"`:** `known` (set by an elder's rumor, or implicitly once `told`) gates whether the creature shows on the world/powers map at all and whether the journal names it; `told` (set only by talking to the creature in person) additionally reveals its full hint and the gear's resting place. So a rumor from an elder only puts the creature on the map — finding the actual gear still means tracking the creature down and hearing its tale yourself. Flow: hear of it (from an elder, or stumble onto the creature's tile regardless of any rumor — `artifact_creature_at` is gated only on `not told`) → roam to the creature → `[e]` **talk** to it (the chat window shows a `creature` branch → `[1]` hear the tale, setting `known=told=True`) → the gear's **resting place now shows on the world map** (gold ✦ + name) → travel there: `enter_tile` puts the tile in **`"guardian"` mode** — a **mini-boss** (`make_guardian`, fac −1, a grey stone warden using the boss AI) amid a **thickened monster watch** (`spawn_monsters` bumps count near an un-found `loc` via `_near_artifact_loc`) → fell the guardian → `claim_artifact` grants the prize (a staff → `staves_found`, equipped if Might allows; or the charge → `can_charge=True`). The world/powers maps mark the creature (violet ●, labelled with name + coords for easy testing) once known, until told, then the `loc` (gold ✦); the journal lists artifact quests under "YOUR OWN QUESTS", with unknown ones folded into an aggregate "N tales yet unheard" line. Helpers: `artifact_creature_at` / `artifact_loc_at`. (Planned: more artifacts/trinkets, creatures that quiz or bargain.)
- **State** — a single `state` dict (`fresh_state()` builds it): `territory[]` (shares summing to 100), `army[]` (0–100), `hearts[i][j]` (a symmetric 0–10 relationship matrix), `items[]` (relics with `{type, owner}`), `owner[][]` (the `GRID`×`GRID` world map — see below), and the wizard/time/favor/town fields (`wizard`, `wiz_stats`, `day`, `world_seed`, `biome_map`; `race[]`, `relic_demand[]`, `next_npc_id`, `searches[]`; `favor[]`, `aid`, `aid_log`; `towns`, `met`, `town_msg`; `shadow`, `corruption[]`, `suscept[]` — see Wizard layer / The fallen god).
- **World map** — `owner[][]` is a `GRID`×`GRID` (50×50 = 2500) lattice of tiles, each holding a power index — the world shown in map mode, that the powers vie over. It is a **pure projection** of `territory[]`, never a second source of truth: `seed_grid()` lays out four contiguous blobs (multi-source BFS to per-power tile quotas) and `reconcile_grid(state, wars)` repaints it each cycle to match `tile_targets()`. It never feeds back into the simulation. A battle's prize is one tile (`TILE_TERR`), so a bigger `GRID` means each battle moves less land — rebalance as the world grows.
- **Alliance obligations** — after drift (step 1b), allied powers (hearts ≥ 8) are pulled into each other's wars and out of them together. When a power crosses *into* war with a third power since last cycle, its whole alliance bloc is dragged to AT WAR (hearts → 3); when it climbs back *out* of war, the bloc is pulled to PEACE (hearts → 5). Propagation is transitive across the bloc and war takes precedence over peace within a cycle. Triggers are detected against `state["prev_hearts"]` (snapshotted at end of cycle), so both drift- and slider-driven shifts propagate.
- **Arrival (the wizard is sent when the shadow claims its first vessel)** — the good god is hands-off until then; `fresh_state()` calls `age_to_danger()`, which fast-forwards the unseen pre-history until `_is_dangerous()`: **one power has fallen** (corruption ≥ `FALLEN_AT`). Since the entity works one vessel at a time, the first fall is the natural alarm and the other powers still stand, so every run begins in a living, contested world (~day 2.5k–4k, all powers still holding real land) with one dark power newly risen. The shadow rises at a brisk `AGE_SHADOW_GROWTH` during the pre-history (stored in `state["shadow_rate"]`), then `age_to_danger()` drops it to the slow playable `SHADOW_GROWTH` and syncs `day = cycle * DAYS_PER_CYCLE`. Capped by `AGE_CAP`.
- **The fallen god (corruption entity)** — the world is made good, but a vengeful presence corrupts it (the good god is hands-off; he sent the wizard). It is modeled as an **agent with a goal**, not weather. `state["shadow"]` (0..1) is its raw presence (grows at `state["shadow_rate"]`/cycle — brisk in the pre-history, slow during play, see Arrival — **plus a snowball term `dominion × SHADOW_FEED`**). Its **goal is dominion** (`dominion(state)` = share of the world under its sway = `Σ corruption × territory`, target 1.0 = the world has fallen, `DOMINION_WIN`). Each cycle it **picks a vessel** — the power with the highest `territory × suscept × (1-corruption)` (`state["shadow_target"]`) — and concentrates its corruption there (`SHADOW_FOCUS` ceiling multiplier) while the rest of the world dims only ambiently (`SHADOW_AMBIENT`). **Corruption is sticky**: it rises toward the pressure ceiling but never recedes on its own (only the wizard, a later slice, will cleanse it), so unopposed the world drifts toward night. Each power's `suscept[i]` (hidden, set at `fresh_state`) sets how far it can fall, so the entity's focus + susceptibility together make some powers fall hard while resistant ones hold near-pure. **Corruption-growth is PLUGGABLE** (`state["corruption_growth"]`, dispatched by `grow_corruption`; `fresh_state(corruption_growth=…)`, default `DEFAULT_CORRUPTION_GROWTH="ambient"`): `"ambient"` is the above (it wells up on its own). `"sown"` is an **experiment** — the shadow does *not* grow on its own; a **sower** (`state["sower"]`, a dark agent / the lord's precursor) **commits to** the realm with the largest *raw* mismatch (`region_mismatch` — army+relic gaps, corruption-independent) until it's sown out, travels to a town or city that realm *currently holds* (preferring its capital; falling back to the nearest tile they hold if they've lost every town), and sows there (`SOW_RATE`), so corruption only takes root where tension is exploited; a region's rising corruption then **swells the GLOBAL shadow** (`state["shadow"] += Δcorruption × SOW_SHADOW_FEED`, sticky), so the shadow's overall presence is a *consequence* of sowing, not weather. Toggle live with **`[o]`** in a dev view (resets into a fresh world of that model); the sower shows as a purple ✦ on the map. **In the tile view he's disguised** (pygame): on whatever tile he's caught sowing, he's folded into that tile's ordinary scenery folk (`Game.sower_npc`, just another wanderer, no hostility) — chatting with him gives the same generic dialogue as any wanderer. Only `WIZ_WITS_SOWER_AT` (6) Wits lets the wizard see through the disguise (`nearby_person`); walking up and pressing `[e]` then **exposes** him (`Game.expose_sower`) — the mask falls away and the tile turns hostile on the spot (the usual sower mini-boss fight, `mode == "sower"`). Below that Wits, or before you've walked up and exposed him, he's just a person — no danger, and no benefit to ignoring or confronting him: only an actual kill triggers `slay_sower`'s respite cooldown, so walking away unexposed or exposed-but-unkilled costs nothing — he simply resumes sowing. Net: ambient runs to dominion ~1 (the world falls); sown is far more contingent (it stalls where no one sows). See the wants-corruption-economy memory. Beyond its own growth, corruption bends the **relationship drift** (step 1): a per-pair skew makes the corrupt belligerent (`CORRUPT_AGGR`, vie for power → hearts trend to war), a corruption *gap* sours relations (`CORRUPT_SPLIT`, the pure vs the fallen), and shared high corruption pulls hearts up (`CORRUPT_BOND`) — but only barely outweighing the belligerence, so two fallen powers *may* ally or may tear at each other (a chance, not a lock; emergent and varied per run). On top of these, **corruption-amplified WANTS** give wars a motive (`_want_pull`, "Layer 1"): a per-pair downward pull from *mismatches* in things factions covet, each side scaled by **its own** corruption (so the pure never covet) — the corrupt-and-relic-**poor** covet the relic-**rich** (relic-wealth = Σ owned relics' `boost`, `CORRUPT_RELIC_WANT`), and the corrupt-and-**strong** prey on the **weak** (`army` gap, `CORRUPT_ARMY_WANT`); bigger mismatch → bigger pull toward war. (Territory is intentionally left out — `desired_share` already covers "they dislike having less.") `want_motive(state, i)` surfaces the strongest current want for flavour (the chat **news** reads "the X covet the relics of / would prey on the weakness of the Y"). *Planned (not built): the lord will actively **prey on** these wants to break the wizard's unities; and eventually the shadow's growth itself becomes an economy of dark agents exploiting wants — see the wants-corruption-economy memory.* Wars/coalitions then emerge from the existing hearts→alliance machinery. It does **not** yet touch combat/territory. Corruption is shown in the **wizard's tile view** too now (the good god's agent senses the shadow): the footer reads out the region's corruption (`% corrupt (band)` for the tile owner) and, on a contested tile, each combatant's corruption. The richer per-faction picture stays in the **dev views**. Both the world map and the controls/slider view carry an always-visible "Corrupting:" strip (`_corruption_strip()`: per-faction %, color-coded, `†`/`‡` — ascii `+`/`*` — marking falling/fallen) so the per-faction picture is never lost below a tall layout. The world map also frays each faction's *territory* by its corruption (`█`→`▓`→`▒`→`░`, via `CORRUPT_BANDS`/`_corrupt_band`/`_gauge`) and the readout shows a Shadow-presence gauge, a **dominion gauge** (its grip on the world, with the current vessel named / "THE WORLD HAS FALLEN" past `DOMINION_WIN`), and per-faction gauges with a `◀ vessel` marker; the controls view exposes a `Shadow` slider and per-faction `corruption` sliders. Both headers show `Shadow %`/`dominion %`. The **controls dev view also draws time-series GRAPHS** along its bottom (`render_graphs`/`_chart`): rolling line charts of per-faction **corruption** (with the `FALLEN_AT` line), **relations** (the 6 heart-pairs, war line at 3), **territory %**, and **army** — so you can watch the macro dynamics (and the wants/corruption→war machinery) play out over time. Fed by `state["history"]`, a `HISTORY_LEN`-capped rolling window sampled every cycle by `_record_history` (so it fills through slumbers/skips too).
- **The lord claims a throne** — the first time a clan's lord goes from rising to **enthroned** (`lord["ever_enthroned"]` flips False→True in `_update_lord`), a one-off announcement fires via `_chronicle` ("The throne of \<town\>, the seat of House \<name\> now belongs to the lord."), surfacing as the usual toast (`town_msg`) in the wizard's tile view and logged in the journal's chronicle. Only the first claiming announces — later returns to a settled throne (`lord["home"]`/`"brood"`) are silent.
- **The lord is not hostile by default** — `is_hostile()` (dominion_app.py) treats the lord's own unit (`make_boss`, `"boss": True`, a real faction index) exactly like any other faction actor: hostile only while `state["favor"][fac] <= WAR_FAVOR`, so a neutral or positive favor leaves him peaceful (he simply stands there — `update_boss` early-returns when `not is_hostile`, no chase/fire/touch; the boss-mode banner switches from "BARS YOUR WAY" to a "BROODS HERE — [e] TO SPEAK" variant). All other boss-AI units (guardian, lord_agent, sower) keep `fac: -1` and stay unconditionally hostile via the same check, so this only changes the lord. While peaceful, `nearby_person()`/`[e]` lets the wizard **talk to him** (a new `lord_npc` chat branch): in the `"sown"` corruption model he can offer the wizard the **same sowing errand the sower himself is on** — accepting sets `state["wiz_sow_target"]` to the current sower's target faction (`state["sower"]["target"]`), and `wiz_sow_tick(state)` (called each cycle from `cycle()`, mirroring `_grow_corruption_sown`'s sow-while-present logic via the shared `_sowable()` check) advances that faction's corruption — at the same `SOW_RATE`-based rate and `SHADOW_FEED` — whenever the wizard physically stands on one of its towns, exactly as if he were a second sower. **Attacking the lord while he's peaceful breaks the peace instantly**: any bolt connecting with his unit sets `state["favor"][fac] = 0` outright (bypassing the normal gradual `KILL_FAVOR_DOWN` decay used for ordinary kills), making him hostile from that moment on.
- **The lord's retinue (sown model only)** — once embodied (`dark_champion`, corruption ≥ `LORD_APPEAR_AT`=0.40) and enthroned at `FALLEN_AT`=0.50, the lord gathers a fixed-size band of `LORD_RETINUE` (7) lesser dark agents (`state["lord_agents"]`, `_update_lord_agents`, called from `cycle()` right after `_update_lord`) — mini-bosses the wizard hunts in person (pygame `mode == "agent"`, a guardian-style fight; `make_agent_boss`). Each agent is either a **secondary sower** (corrupts a realm the main sower isn't working, picked by `region_mismatch`, at `AGENT_SOW_RATE`, a fraction of the main sower's rate — several agents CAN still pile onto the same realm, no claim-exclusivity here) or sent to **reinforce a war front** when the lord's own power is under pressure (lending `AGENT_FRONT_STRENGTH` via `_agent_strength` into `effective_strength()`) — never both, and never promotable to lord himself (only `dark_champion`'s pick of the most-corrupted power does that). Killing one (`slay_lord_agent`) just frees a slot; recruiting a replacement takes `AGENT_RECRUIT_CYCLES` (180, ≈5 years — slow, unlike the lord's own swift rise). Gated entirely to `corruption_growth == "sown"` — in `"ambient"` mode `lord_agents` stays empty. Pygame shows them on the world map as a violet dot (alongside the sower's purple ✦). (Don't confuse this with the *wizard's* own retinue — his recruits, below — which has its own, separate capped-party mechanic.)
- **War is corruption-gated; powers fight for a *desired share*** — wars are no longer random. **The war zone (hearts ≤ 3) is walled off by a floor:** in the drift step `_heart_floor()` stops a pure pair from ever dropping below 4 (ENEMIES), so the uncorrupted never descend into war on their own. Corruption removes the floor for a fallen power (corruption > `FALLEN_AT`), so *its* hearts can slide to ≤ 3 — and war then fires automatically through the usual `status_of` ≤ 3 path (the alliance bloc, step 1b, respects the same floor: a pure ally is pulled only to enmity, not war). How much land a side fights for is `desired_share(i)`: the uncorrupted want only an even split (`100/N`) — they fight solely to *reclaim* what they've lost, never to expand, and stop at their fair share; past the fallen line ambition scales up to wanting the whole world at full corruption. Battles clamp land taken to `desired_share - territory` (both at open and per-cycle in 2b), so a pure winner reclaims to equilibrium and halts while a fallen winner presses on. Net: a pristine world is at peace; the shadow is what brings war (and since it always rises, peace never lasts).
- **Simulation** — `cycle()` advances one turn through fixed steps: the fallen god stirs (step 0: grow `shadow`, drift `corruption`) → relationships drift (corruption-skewed, war-zone floored) → wars (hearts ≤ 3, only reachable once corruption unlocked the floor) resolve via `effective_strength()` (army × land × luck × ally support × relic boost) → armies regrow toward what their land sustains → territory renormalizes to 100% → seekers/the lord's hunt may turn up relics (relics **no longer spawn from the world on their own** — they enter play only by being *sought*) → `reconcile_grid()` projects the new territory onto the map. A war whose relative-strength gap is below `STALEMATE_GAP` is a **stalemate**: both armies bleed but no land moves. A decisive war does not flip land at once — it **opens a battle** (`state["battles"]`) over a **single tile** (`TILE_TERR`, clamped to the winner's `desired_share`): the winner is decided up front, then step 2b transfers that one tile's worth of territory gradually over **1–20 cycles** (`rate` = `TILE_TERR` / duration; duration scales inversely with the winner's army *share*, ±40% luck — so the army gap sets the speed, not the prize). Only one battle exists per power-pair at a time; a sustained war just opens a fresh one-tile battle once the previous tile falls. A battle runs to completion even if the war ends diplomatically, and ends early if the loser hits its territory floor. The per-cycle slices populate `wars` for `reconcile_grid`. Relics are looted by the stronger side every cycle of war, stalemate or not. **Wizard aid** (`state["aid"]`, set for one cycle): the backed side gets `+AID_BOOST` effective strength (so it wins clashes / opens battles in its favor); in an in-progress battle, aiding the **winner** multiplies its transfer rate (`AID_RATE_MULT`), while aiding the **loser** stops land moving and harries the attacker — after `AID_FLIP` straight cycles the battle's winner/loser **swap**. `aid` is cleared at end of cycle; `settle_aid()` then pays out `favor` for any ended engagement. When a battle ends (tile fully taken, or loser floored), `_ended()` posts a result toast (`state["town_msg"]`) **only if the wizard's current tile owner was a combatant** — so the player sees who won the fight on their front, not every battle worldwide.
- **Wizard layer** — a thin first slice of DESIGN.md's player avatar. `state["wizard"]` is a macro-tile position (the only genuinely new player state). The **tile view** is the wizard's lived, default view: he stands in a `TILE_GS`×`TILE_GS` (20×20) micro arena inside his current macro tile, and it is now **realtime** (a Zelda/NES action layer — see Realtime arena). **Time is wall-clock while inside a tile:** `SECONDS_PER_DAY` (20) real seconds spent on a tile = one world-day, and that elapsed time is **banked on exit** — stepping off an edge into the next macro tile runs `advance_days(state, ceil(seconds / SECONDS_PER_DAY))` (always ≥ 1 day; round up). The macro `cycle()` sim is **frozen while he is inside a tile** (it ticks only on the boundary cross, or on an explicit world-turn — slumber/step/aid — each of which also restarts the wall clock, `tile_enter`). All world-time still routes through `advance_days(state, n)`, which fires one `cycle()` per `DAYS_PER_CYCLE`-day boundary it crosses. **Walking off an edge carries the wizard into the adjacent macro tile** (clamped at the world border), re-entering him on the opposite edge; his local arena position (`tile_lr`/`tile_lc`) and the whole realtime-arena substrate (monsters/shots/facing/cooldowns/`tile_enter`) are transient main-loop state, not stored in `state`. The **world map** (the 50×50 macro overview, rendered as a fit-to-screen minimap) and the **controls** sliders are *dev views*, reached by cycling `m`; on the dev map a step crosses a whole tile, so it advances 10 days (one cycle). Step (`s`) advances one cycle (10 days) via `advance_days`; **auto-play is gated to the dev views** (it makes no sense in the realtime arena, where time is the wall clock). **Wizard agency (first slice):** on a contested tile, `1` backs the tile owner and `2` backs the enemy — each press sets `state["aid"]`, passes one cycle, and is tallied in `state["aid_log"]` (per sorted power-pair). When that engagement ends (no active battle and either peace returned or a battle the wizard fought in resolved), `settle_aid()` moves `state["favor"][i]` (each faction's 0–10 standing toward the wizard, neutral 5, **never drifts** — only the wizard's deeds change it): the side he backed warms by up to `AID_FAVOR_MAX`, the other sours. Favor is displayed but not yet consumed by other mechanics. **Towns** (`place_towns()` → `state["towns"]`, keyed by `(r,c)`): each faction gets a **capital = a CITY of up to 4 tiles** (a 2×2 block of its own land; the anchor tile carries `"main": True` and the keep, the other 3 are districts, all `capital: True`) + ~9–15 villages (`VILLAGES_PER_FACTION`, spaced ≥2 tiles), chosen once from `world_seed` and **fixed to the land** — a town's faction is the founding populace and does not change when the tile is conquered. The pygame map draws each capital tile as a white-ringed faction square (the 2×2 reads as a city) with the name on the `main` tile. **Tile-view ambient life** (`Game.tile_scenery`, pygame-only, deterministic per `(world_seed,r,c)`): a peaceful, town-less, un-fallen tile is dressed by its distance to the nearest town (`OUTSKIRTS_RADIUS`) — right beside a town it's a hamlet (a house or two, farms, a couple of folk), within a few tiles a lone house/farm + a soul, deep country just the odd wanderer; fishers appear wherever the biome is the Azure-realm coast. Folk (`[x,y,vx,vy,kind]`) wander the clearing in realtime. Fallen tiles stay menacing (monsters, no civilians); town tiles use the recruitable crowd instead. This is to make the wizard (tile) view worth roaming. A town tile's micro scene (`tile_town()`) lays buildings (keep/houses) and townsfolk over the wilderness, deterministically per run. Peaceful town: townsfolk shown; walk up and press `e` to **talk** (the chat window — see Knowledge of the world; greeting no longer grants favor). Contested town: townsfolk flee and the besieging hosts are drawn instead. Towns are marked on the dev map (`TOWN_MAP_GLYPH`). Still unbuilt from DESIGN.md: deterministic micro *generation* beyond terrain/towns, epoch caching, detach/reattach, overrides, NPC behavior/movement, and favor feeding back into faction behavior.
- **Realtime arena & monsters** — the tile view is a realtime Zelda/NES action layer (see the time model under Wizard layer). A **fallen** macro tile (its owner's `corruption ≥ FALLEN_AT`, and the tile peaceful + townless) is infested by the shadow's minions: `spawn_monsters(state, r, c)` places `1..MONSTER_MAX` **stationary LOS sentries** (👹 / `M`, red), deterministically per `(world_seed, r, c)` (so a tile always lays out the same), count scaling with the owner's corruption. Each frame a sentry that shares the wizard's row or column fires a **slow** projectile (`MONSTER_PROJ_SPEED`) down that axis on a `MONSTER_FIRE_COOLDOWN`; the wizard fires a **fast** bolt (`WIZ_PROJ_SPEED`, `WIZ_FIRE_COOLDOWN`) in his facing dir with **space**. A bolt hitting a sentry kills it (no respawn until the tile is re-entered); a sentry's shot hitting the wizard costs **one heart**. Monster cells are impassable (you bump them). Monsters are a **pure action/hazard layer — they do NOT feed the sim** (no corruption cleanse, no territory effect): fallen land is simply dangerous to cross, and clearing it is optional. `state["health"]` (0..`WIZ_HEARTS`=10, ♥/♡) is the wizard's life; at **0 hearts** movement/firing are blocked until he slumbers. Projectiles (`{r,c,dr,dc,src}`, `src` = `"mon"`/`"wiz"`) and monsters live as main-loop locals, advanced in a per-frame physics block in `main()`; the macro sim stays frozen during all of this.
- **Wizard upkeep (energy & life)** — player-facing (shown in the tile view; unlike corruption it is not hidden). `state["energy"]` (0..`ENERGY_MAX`) is spent aiding battles (`AID_ENERGY` per cycle of aid; aiding is blocked when too low); `state["health"]` (0..`WIZ_HEARTS`) is spent to monster fire. Energy is restored by eating a `food` ration (`f` → `+FOOD_ENERGY`, instant, no world-time); **slumber** (`z`) opens a **menu** (`self.slumber`, `handle_slumber`/`render_slumber`): **[1] rest until hale** (`Game.rest` — the old behaviour: recover energy+life to full at the cost of the deficit's days, or, if the age is won, `skip_to_next_age`), or sleep a chosen span — **[2] 50y / [3] 100y / [4] 1000y** (`Game.long_slumber` → `slumber_years(state, years)`): the world and its bloodlines run on while he sleeps and he wakes hale to a changed (or fallen) world. `slumber_years` fast-forwards `years×DAYS_PER_YEAR`, calling `cycle(state, reconcile=False)` for most cycles (the territory→map projection is skipped while no one's looking — borders lag, are repainted every ~30 cycles and once exactly at the end) so even a 1000-year sleep stays ~6–12s; lineage/shadow advance every cycle as normal. Deeds that happen while he sleeps (seekers finding/losing/dying, successions) are kept in a rolling **`state["chronicle"]`** (`_chronicle` — sets `town_msg` *and* logs `(year, msg)`), so on waking he's told "N deeds passed" and can read them in the journal's **CHRONICLE** section, rather than the news being silently overwritten. **Seekers — the stat/journey system** (now embodied as **Recruits**, `state["recruits"]` — see the Recruits bullet; the curses modal flow below is legacy): in a peaceful town, `k` opens a two-step modal (handled by `commission` in `main()`): **read the crowd** (each townsperson's rolled stats, drawn in the tile view) → **pick a person** → **pick a relic-journey**. Dispatch costs `SEARCH_ENERGY` and **promotes** that NPC (snapshot stats + a numeric `id`; the relic goes to the town's own faction). The errand plays out off-screen: each cycle (step 5b) a progress bar (`prog`, 0..100) climbs by `seek_rate(fit, base)`, where `fit = journey_fit(stats, relic_demand[type])` measures how well the seeker's stats meet the journey's per-stat demands and `base = relic_base_cycles(type)` is the relic's inherent length (scaling with its power). `ETA = base / fit**SEEK_FIT_POW`, clamped to `[base, SEEK_USELESS_CYCLES]`. So the **easiest** relic at a perfect fit ≈ 6 months (`SEEK_FAST_CYCLES`=18) and never worse than a few years; the **mightiest** is a legitimate ~10-year quest even perfectly crewed (`SEEK_HARD_CYCLES`=365), stretching toward a **useless ~100 years** (`SEEK_USELESS_CYCLES`=3650) with the wrong person. If a rival (or the lord) gets that relic first, the errand **fails** ("in X's hands"). The tile view lists active seekers with a progress gauge + ETA. The crowd is **re-rolled every visit** (`roll_townsfolk`, transient main-loop state — DESIGN.md's anonymous→promoted ladder); the lord hunts by the very same `fit` math (his own stats), so the relic race is a contest of fit. Food is bought (`b`, `FOOD_PRICE` gold) from a town market or a travelling pedlar: there is no wandering merchant entity — `state["pedlar_here"]` is a random roll (`PEDLAR_CHANCE`) made each time the wizard enters a new macro tile. `merchant_here()` reports `'town'`/`'pedlar'`/None. Gold has no income source yet. Transient feedback uses `state["town_msg"]` (shown on the tile view's row 1).
- **Recruits (persistent characters) — the WIZARD's own retinue** — `state["recruits"]` (engine: `add_recruit` / `_update_recruits` in `dominion.py`; UI: pygame only; not to be confused with the **dark lord's** retinue, `state["lord_agents"]`, a separate cast — see above; the legacy curses front-end's `k` commission instead dispatches into the older, simpler `state["searches"]` seeker list and is unaffected by any of this). Each dispatched seeker is now an agent `{id, fac, race, stats, r, c, task, relic, fit, prog, target, party, party_of, given, house, born, lifespan}`, capped at `MAX_RECRUITS` (7), and displayed by **name** — `d.recruit_name(q)` = `"Given of House X (#id)"` (or just `"Given (#id)"` with no house) — rather than a bare id, used wherever space allows; tight HUD/overlay contexts fall back to just the given name. Lifecycle, advanced each cycle (step 5b): a recruit can be commissioned straight onto a journey, or **recruited idle** (`task: "idle"`, `relic: None`, `fit: 0.0`) — they simply join the wizard's following at his tile and wait, doing nothing and risking nothing (`_update_recruits`'s task dispatch and the seek/march-only peril roll both skip an idle recruit) until ordered. From idle (or any state) **seek** (roam the map, tick `prog` by fit — a rival grabbing the relic ends the seek) → on success the relic goes to `fac` and they **march** to the nearest war front (`_front_tile`) → **fight**, lending `RECRUIT_STRENGTH` to their faction's `effective_strength` (`_recruit_strength` — the macro "unite the peoples" lever; they fight for their **home faction**, directing them to any faction is a later step). Pygame UI: the `k` commission modal's relic-pick step offers **`[0]`** to recruit idle instead of picking a journey; the `t` reassign modal offers **`[5]` stand by** to revert an already-active recruit back to idle (clearing `relic`/`prog`/`target`, and dropping `led` if they were being led). **Shared quests (parties):** `join_quest(state, follower_id, leader_id)` puts one recruit onto another's seek-quest, capped at `RECRUIT_GROUP_CAP` (`round(MAX_RECRUITS * 0.3)` = 2) members total per quest — a party, not a race: only the leader's `prog` advances, followers shadow the leader's position (`party_of`) instead of seeking independently, and the leader's `fit` is swapped for `quest_group_fit` — the elementwise **max** of the whole party's stats scored against the relic's demand via `journey_fit`. So a party member who duplicates a stat the party already covers barely raises the fit; one who fills a stat the party lacks (e.g. someone has high Might already — adding another high-Might recruit does little, but adding a high-Wits one does) raises it more. `leave_quest(state, recruit_id)` disbands a party (a follower resumes seeking independently with fresh progress; if the leader leaves, all followers are freed too). *Deferred (not implemented):* once NPCs/agents can themselves fall to corruption, party-mates might turn on each other on a shared quest instead of always cooperating. They roam visibly so the wizard can **catch** them; pygame draws them on the map (faction dot, gold ring = fighting) and materializes them in their tile, where **`t`** opens a reassign modal (march ↔ seek-a-relic; sending a seeker to battle forfeits its `prog`). They persist — only the wizard reassigns them, **or a beast slays them**: a traveling recruit (seek/march) risks a monster encounter each cycle (`recruit_peril` × `RECRUIT_ENCOUNTER`), resolved by **Might + Endurance** vs a shadow-scaled threat → shrug off / wounded (lose `prog`) / **slain** (removed, `RECRUIT_BEASTS` flavor). **`recruit_peril` is now dominated by `quest_difficulty(state, q)`** — a 0–1 figure from the relic's inherent hardness (`relic_base_cycles`) blended with the seeker's lack of fit, with the land's corruption/shadow only a minor ambient add. The *same* `quest_difficulty` figure sets find-odds and the lead-journey's monster density (macro % → wizard view). **Meet-them-in-battle** (pygame `enter_tile`/`update`/`cross`): catching an agent in open country rolls `recruit_peril` per agent — a **beset** one (marked `BESET!`, red) has monsters on it *now* (`self.beset_ids`). Clear every beast → **saved** (seekers get a progress surge); cross away and you **leave them to it** (`_recruit_encounter`). 
- **Leading a seeker in person** (the attention pillar; pygame, seekers only): the `t` orders modal offers **`[3] lead / part ways`**. While `q["led"]` is set, the seeker travels with you (their tile tracks the wizard at walking pace) and keeps ticking `prog`, but their abstract per-cycle peril roll is **suppressed** (`_update_recruits`) — the danger is realized as **real combat**: `enter_tile` spawns a monster watch (and, on hard roads, a road **warden** = `make_guardian(None)`) sized by `quest_difficulty`, and adds the seeker as a fighting **ally** (`make_ally`: auto-fires at monsters, **Endurance hearts** as `q["hp"]`, friendly-fire-immune). Enemy shots cost the ally hearts. At **0 hearts**, `ally_down` rolls vs **Endurance**: high ⇒ **downed** (`q["downed"]`, out of the fight) — **camp `z`** revives them to full (and banks days); low ⇒ **slain**. **Crossing away while a companion is downed kills them** (`cross`). **Part ways** (`[3]` again) clears `led`; the seeker resumes roaming, progress intact. `led_seeker(state)` finds the escorted one; one at a time. (Still later: leading marchers/fighters; "fight for any faction".)
- **Lineage (houses, rulers, bloodlines)** — the world's notable people **age, die, and beget heirs continuously** as time passes, so the wizard (who endures across ages) watches generations turn over. Per-race **age curves** `RACE_AGE` (years: Elves ~2400–6000 + slow maturity — a near-immortal throughline; Men ~58–95; Dwarves ~190–320; Cyclopes ~70–130), `DAYS_PER_YEAR=360`. **Two populations, two treatments** (the macro%→micro ethos): the **tracked notables** (`state["notables"]` — the four royal **houses**, `state["houses"]` fac→surname, each a ruler + kin, named from `HOUSE_NAMES`/`GIVEN_NAMES`) evolve every cycle via `_advance_lineage` (age by `_now(state)=cycle×DAYS_PER_CYCLE` — which advances even in the day-stale `age_to_danger`/`skip_to_next_age` loops, so a century slumbered = a century of turnover; death at lifespan; capped births `_child_of` with `inherit_stats` = parent regressed-to-race-mean + noise, *no fading decay yet*; **`_succeed`** seats the eldest living kin when a ruler dies — princes inherit, thrones never empty); the **anonymous crowd** is generated **lazily** by `roll_townsfolk` (living house kin — recruitable, even a prince — padded with fresh commoners), so after a skip the faces are the *descendants* of those you knew. Recruits are notables too: `add_recruit` carries `given/house/born/lifespan`; in `_update_recruits` they can **die of old age mid-quest**, leaving an heir in `notables` (the line continues — Bilbo→Frodo). Helpers: `ruler_of`, `house_of`, `person_age`. **Kings are face/lineage-anchor only** — they do NOT move territory/army yet. `found_houses` runs in `fresh_state` before the pre-history. Surfaced (pygame): the recruit modal shows names/ages/houses (◆=noble), the world/powers map labels capitals with their house, chat **news** names the reigning house, recruits' toasts use their names.
- **World geography (the wizard's traversal terrain)** — a persistent, **connectivity-first** terrain layer at **micro resolution**: one cell per tile-view cell, so the whole world is a `(GRID*TILE_GS)`-square lattice (**1000×1000**). Generated once per run by `generate_geography(state)` (`dominion.py`, from `fresh_state` after `place_towns`); stored as a **`bytearray`** `state["geo"]` of `GEO_*` cell codes (`GEO_LAND`/`GEO_WATER`/`GEO_MOUNTAIN`/`GEO_ROAD`/`GEO_BRIDGE`/`GEO_PASS`, row-major) + `state["geo_w"]`. **Rivers** are thin (~2-wide) bands that wind across the world; **mountain ranges** are masses; a **road network** links every town through **BRIDGE** (over a river) and **PASS** (through a range) crossings. Algorithm (same as the macro version, at micro scale): plan a road graph (MST + loop edges) over the town nodes (capital keeps + villages, at their tile centres) *first* → scatter rivers/ranges freely (never on a town's protected tile) → **realize each road by carving through whatever blocks it** (river→bridge, mountain→pass). **Connectivity is guaranteed by construction** (every planned road is a continuous passable corridor between its two towns), so **no runtime flood-fill** is run (`_geo_flood` exists for headless tests only). Generation is ~0.2s; deterministic per `world_seed`. `geo_cell(state, x, y)` reads a cell; `tile_walkable(state, r, c)` = "the tile's 20×20 window holds any passable cell" (used for the wizard's seat + crossing guard; almost always true now that tiles are mostly land). **It is the wizard's *traversal* layer ONLY — it does NOT feed the macro sim** (territory/owner/war/recruits untouched; the curses build ignores it via `geo is None`). Spike/algorithm origin: `spike_pygame.py`. Pygame wiring (`dominion_app.py`), **two consumers**: (1) the **tile view** — `build_geo_terrain()` (run in `enter_tile`) slices the 20×20 **window** of the global grid at `(wr,wc)` into a collision mask `self.tile_solid` (water/rock) + a paint map `self.geo_paint`; because it's one slice of a single continuous grid, rivers/roads **line up across tile edges for free**. `solid_at()` consults `self.tile_solid` (the **only** thing that collides — decorative `tile_terrain` stays passable); units (monsters/soldiers) and the wizard's spawn are kept off solid cells (`nearest_open`); `cross()` is gated by `tile_walkable`. (2) the **world map** — the map view has two modes (`self.map_mode`, toggle **`g`**): `render_world_geo()` blits a **cached** Surface (`bake_world_surf()` renders the 1M-cell grid to a `Surface` via a colour LUT + `frombuffer`, scaled to the arena, re-baked only when `world_seed` changes) and `draw_map_markers()` is shared with the **powers map** (faction/corruption/shadow). Both unlock when you buy a map. `GEO_PAINT_RGB` (tile paint) / `GEO_MAP_RGB` (world map) hold the colours. Known polish gap: the cosmetic `tile_terrain` wall-ring glyphs can read like geo mountains on a busy tile.
- **Rendering** — `draw()` (controls), `draw_world()` (the macro world as a **fit-to-screen minimap**: each displayed cell summarizes a `block`×`block` patch of tiles — `block` chosen so the whole `GRID`×`GRID` world fits the terminal — by majority owner, corruption-shaded; wizard patch + towns take precedence; readout shows Shadow/dominion/per-faction gauges below), `draw_tile()` (the wizard's NES-overworld view of one macro tile). Every tile renders as a bordered clearing of biome terrain framed by a jagged wall ring; a **contested** tile (`tile_enemy()` finds a neighbour at war) draws the two hosts of soldiers **only while an active battle rages** there (they withdraw when it's won); the footer shows a battle-status line — who's taking the tile, a progress gauge + cycles-left, and a "defenders holding N/`AID_FLIP` to turn the tide" counter when the wizard is shoring up the underdog (or "no battle rages" when the war is in a lull); a **town** tile (`tile_town()`) overlays buildings and townsfolk on that terrain (and shows the besiegers instead when contested). Terrain (`tile_terrain()`, `BIOMES`, `_ground_cell`/`_feature_cell`) is a deterministic cosmetic projection that **never feeds the sim**. It is **fixed land**: the biome (Crimson→ashen rock, Azure→coast/pond, Verdant→wood, Gilded→sand) is snapshotted at start into `state["biome_map"]` from the *initial* owner, so conquering a tile does not re-theme it. A per-run `state["world_seed"]` makes the world random each run; terrain regenerates deterministically from `(world_seed, r, c)`, so a screen is identical whenever the wizard returns and the whole world persists for the run (only a seed + a `GRID²`-int biome map are stored, not the cells). Terrain uses a 256-color palette (`init_terrain()`, `TERR`) with an 8-color fallback; pairs are allocated lazily via `_cpair()` (a `(fg,bg)`→pair cache starting at 50). Glyphs come from `GLYPHS` in two tiers (`ascii` / `fancy`), chosen by UTF-8 detection and toggled live with `g`. In the tile battle view each faction's soldiers are a distinct human emoji (`FACTION_EMOJI`, fancy) — emoji can't be tinted by curses, so the two hosts are told apart by avatar + position (enemy upper ranks, owner lower) rather than color; ASCII mode falls back to colored faction initials. The macro minimap stays color-coded blocks (tintable). **Pixel-art textures (pygame, hybrid swap)**: a handful of CC0 textures from Kenney's "RPG pack" (`assets/kenney_rpg/`, see `LICENSE.txt` there) replace the flattest procedural drawing — grass/dirt ground fill, the pond/water cells, and the tree (`♣`) feature glyph now blit a 32×32 texture instead of pygame primitives (`load_textures()`, `Game.tex`, wired in `render_tile`/`draw_feature`). Everything else — wizard, NPCs, monsters, soldiers, buildings, UI/HUD, the rock ground kind, and the peak/boulder/flower glyphs — stays procedural by design; texture loads fail soft to `None` so a missing asset file falls back to the old drawing rather than crashing.
- **Loop** — `main()` is a non-blocking (`nodelay`) ~50fps `getch` loop, starting in the **tile** view. It runs the realtime-arena physics each frame in tile view (monster fire / bolts / projectile resolution), then an auto-play tick gated to the dev views. `m` cycles the three views (tile → world map → controls → tile; re-entering tile respawns monsters + restarts the wall clock), `g` toggles glyph tier, **space** fires a bolt in tile view (and toggles Play in the dev views). Arrows walk the wizard freely within the arena and across tiles in realtime — banking elapsed seconds as days on each edge cross (tile) / walk the wizard one tile = 10 days (dev map) / navigate sliders (dev controls). Step (`s`) advances 10 days (one cycle); explicit world-turns (`s`/`z`/aid) and edge-crossings all route through `advance_days` and restart `tile_enter`.

### Key invariants when editing

- `state["hearts"]` must stay symmetric: any write to `hearts[i][j]` must mirror to `hearts[j][i]` (see the `adjust()` handler and the drift step).
- `territory` always sums to ~100; `cycle()` step 4 clamps each entry up to `MIN_TERR` (1.5) and *then* renormalizes to 100%. Because renormalization divides by the post-clamp sum (which exceeds 100), a power sitting at the floor can be scaled marginally below `MIN_TERR` (observed low ≈ 1.483) — the floor is **soft**, not strict. The territory slider redistributes its delta across the other powers.
- Relics **enter play only by being sought** — the wizard's seekers (`cycle()` step 5b) and the lord's hunt (`_update_lord` → `_grant_relic`); they no longer well up from the world at random (the old `spawn_item` random-spawn is gone). Once in play, a relic changes owner as **spoils of battle** (the stronger side seizes the loser's relics) — this fires in *every* war including stalemates and even when no land moves, so relics are more volatile than territory. Max `MAX_ITEMS` (3) relics exist at once; a new find past the cap eclipses the weakest existing relic.
- The world map mirrors `territory[]` and nothing else. After `cycle()`, tile counts equal `tile_targets()` exactly (verified to 0 drift over 90k cycles). War flips are clamped so no loser drops below its target — a tile-less power has no border and could never recover — and a chaining cleanup pass settles the rest. Winners stay contiguous; a hard-hit loser may fragment (cosmetic, accepted).
